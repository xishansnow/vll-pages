Block A* 算法

Created2025-01-17|Updated2025-01-23|任务规划单机规划
|Word Count:1.6k|Reading Time:5mins

Block A* 是一种面向网格地图的路径规划算法，通过将网格划分为更大的 “块” （block），在块级别进行快速跳跃或查找，从而减少搜索节点数量、加速路径规划。它常与 JPS（Jump Point Search） 等方法相结合或比较，用于大规模离散网格上加速 A* 搜索。下面从原理、算法流程、优缺点、示例思路等方面进行介绍。

一、算法原理

  1. 分块（Blocking）
    • 将原先的细颗粒网格（例如 1×1 每格）聚合成更大粒度的 “块” （如 4×4、8×8 等），在块上构建抽象化的搜索图。
    • 每个块包含若干网格单元，并为块的边缘选取 “关键信息” ——例如可通行的边缘格子或预计算边缘到边缘的最短路径。
  2. 块级跳跃
    • 当进行 A* 搜索时，如果当前位置可通过预先计算好的信息迅速判断跳跃到某个块的另一个边缘位置，则无需遍历块内部的所有节点。
    • 这类似于预处理一个块内的通行情况，一旦确定能在块内从某边缘到另一边缘直通，就跳到块边缘，不用在块内离散搜索。
  3. 与 HPA* 的区别
    • Block A* 的块通常规模更小，更精细，并且主要关注预处理块内通行和与相邻块的接口；
    • HPA* 则更像多层次抽象，会在更高层级（ “簇” ）进行出入口点抽象搜索。

二、算法主要步骤

  1. 网格分块
    • 例如将 \(N \times N\) 的网格按 \((blockSize \times blockSize)\) 分成 \(\frac{N}{blockSize} \times \frac{N}{blockSize}\) 个 Block。
    • 在每个 Block 内部可进行预处理，记录块边缘之间的连通情况和最短距离。
  2. 预处理
    • 对每个 Block 的边缘格子进行通行分析，如果块内部是全部空地，可直接设置 “可直达” ；若块内存在障碍，则需要本地 BFS/A* 查找。
    • 将块间边缘格子连接成更小的抽象图，以供 A* 搜索时快速跳跃。
  3. 搜索
    1. 在全局 A* 中，状态可被抽象为 (block_id, edge_cell)；
    2. 当从一个 (blockA, edge_cellA) 转移到相邻块 (blockB, edge_cellB) 时，可直接引入预先计算的代价（块内或块间边缘到边缘的最短距离），无需枚举块内所有格子。
  4. 路径还原
    • 当搜索结束得到抽象路径，需要在每个 Block 内拼接本地最短路径以获得完整网格路径。

三、优缺点

  1. 优点
    • 减少搜索节点：A* 主要在块的 “边缘级别” 进行搜索，能显著减少状态数量。
    • 加速多次查询：对同一地图多次路径规划时，预处理可复用，加速后续查询。
  2. 缺点
    • 需要预处理：需要对每个 Block 内部及与相邻 Block 的连通进行存储，对内存有额外开销。
    • 路径次优：由于存在块级抽象，可能导致的路径不如全细粒度搜索精确，但通常误差很小。
  3. 适用场景
    • 大规模网格地图，且要多次路径查询时优越性明显。
    • 单次查询在小地图时可能不如传统 A* 更简单直接。

六、总结

Block A* 通过将网格地图分块，并在块内预处理和在高层抽象图上运行搜索，从而减少在原始网格中进行大范围搜索的开销。该方法适合在地图多次查询时复用预处理结果，从而实现高效的路径查询。
