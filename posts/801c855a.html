压缩路径数据库（ CPD ）算法

Created2025-01-17|Updated2025-01-23|任务规划单机规划
|Word Count:2k|Reading Time:7mins

CPD（Compressed Path Database） 算法是一种针对网格地图（或其它离散图）进行多次路径查询时高效检索的技术。它由 [Strasser et al., 2017] 等人提出，被广泛应用于游戏AI、导航系统等场景下的重复路径查询（Offline Query）。

CPD 的核心思路是：
1. 在离线阶段对地图进行预处理，构建压缩后的路径数据库（Compressed Path Database）；
2. 在在线阶段（实际查询时）通过查表或少量运算即可快速还原从起点到终点的最短路径。

下面从原理、主要流程、优缺点以及示例实现思路进行介绍。

一、CPD 算法基本原理

  1. 离线预处理阶段：
    • 构建最短路树 / 图：对地图中的每个节点（网格坐标）进行最短路径计算，如 BFS 或 A*，或将其划分为多个区块后在抽象图上做全局 Floyd-Warshall 等；
    • 压缩存储：以“方向向量”或“中间关键节点”的方式对最短路信息进行存储和压缩。
    • 具体做法各有不同，但目标是让从任意节点 s 到任意节点 t 的最短路径可在数据库中以较小存储量记录或复原。
  2. 在线查询阶段：
    • 用户给定起点 start 和终点 goal；
    • CPD 根据预处理得到的压缩信息，在常数或对数时间内查找出路径要经过的关键节点或方向，从而迅速恢复完整路径。
    • 大量重复查询时，CPD 能比重新用 A* / BFS 等显式搜索快很多。

二、主要流程示例

  1. 构建最短路信息
    • 对于图中每个节点 v，需要记录或能推断出通往其他节点的最短路径信息；
    • 直接存储所有节点对 (v, w) 的最短路径会非常庞大（可达 O(N^2)），因此进行压缩；
    • 典型压缩方式之一：
      • 对地图做分层或分区；
      • 或用最短路树加“跳点”记录等结构减少冗余信息。
  2. 生成 “CPD”
    • 在压缩路径数据库中，对于每个节点 v，可以存“下一步方向”或 “下一关键节点”以到达目标 t 的信息，取决于路径选择策略；
    • 通过分块、边界节点等加速存储。
  3. 查询过程
    • 给定 (start, goal)：
      1. CPD 查表：在 start 节点中读取“前往 goal 时下一步走向何处？”；
      2. 移动到下一个节点后，再查一次“从该节点去 goal 的下一步是哪里？”。
      3. 反复重复，直到到达 goal。
    • 若路径中可能会遇到障碍变化或新的动态约束，则需要更新CPD或采用局部搜索修正。
  4. 优点
    • 多次查询时非常快；
    • 占用的内存比完整存储所有最短路径要小得多。
  5. 缺点
    • 预处理开销大：在大型图中建立 CPD 需要较长时间；
    • 存储依然可观：虽然已压缩，但对超大地图依然可能占用大量空间。
    • 动态环境：如果障碍变化，需要更新或重建数据库。

三、CPD 在网格地图上使用的示例思路

以下提供一个简化版的示例伪代码，以说明 CPD 构建与查询的大致流程。请注意，真实实现中，CPD 的构造、压缩策略可比此更复杂。

1. Offline BFS/A*

  • 在离线阶段，对每个节点 n 进行 BFS/A* 以获取 n->t 的最短路径 dist/nextNode。
  • 若地图节点数很多，可以合并相邻节点、使用分块或跳点技术进行压缩，这就是Compressed Path Database 的核心。

2. Online Lookup

  • 在查询 (start, goal) 时，从 start 开始反复查表： CPD[current][goal] 给出下一个节点，然后跳到下一个节点，直到到达 goal。

四、应用与改进

  1. 游戏AI：
    • CPD 在 RTS 游戏地图、RPG 地图中常被用于大量重复查询场景，比如多个角色轮流寻路。
  2. 路径平滑：
    • 得到离散网格路径后，可做线段合并或平滑处理。
  3. 动态图：
    • 当地图有障碍变化时，需要更新受影响部分的 CPD，否则查询结果不再有效。
  4. 结合分层：
    • 与 HPA* 类似，可在较高抽象层中使用 CPD, 以进一步减少内存和加快查询。

六、优缺点

  • 优点
    • 在地图上进行多次查询时，每次只需 O(pathLength) 级别查表即可恢复完整路径，非常快速。
    • 相比存储所有最短路径，CPD 的存储量被压缩。
  • 缺点
    • 构建时间 和 存储开销 在大型地图时仍然庞大，需要更先进的拆分、分层或区块技术。
    • 若地图变化，需要部分更新 CPD；若变化频繁，CPD 收益不明显。

结论

CPD（Compressed Path Database）通过离线的全图预处理，将网格节点之间的最短路径信息“压缩”存储为 (当前节点)->(目标节点) 的“下一步”映射，从而在在线查询时能以 常数或对数时间复杂度 构造最短路径，用于多次查询场景非常高效。
